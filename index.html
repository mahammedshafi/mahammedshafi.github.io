// Jenkins · Maven · AWS EC2
END-TO-END
CI/CD PIPELINE

A fully automated software delivery pipeline that takes code from a GitHub commit all the way to a live deployed application — with zero manual steps.

JenkinsJenkinsfileMavenNexus RepositoryApache TomcatAWS EC2Git WebhooksIAM
↑ 35% reduction in manual deployment effort
// What I Built & How It Works
  • Designed a declarative Jenkinsfile pipeline with Master-Slave architecture — master node handles orchestration, slave nodes execute the actual build and deploy jobs
  • Configured Git Webhook triggers so every push to GitHub automatically fires the pipeline — no one needs to click "Build" manually ever again
  • Used Maven to compile the Java application, run tests, and package it into a deployable artifact (.war file)
  • Nexus Repository Manager stores versioned artifacts — so you can roll back to any previous build if something breaks
  • Final artifact is deployed to Apache Tomcat on AWS EC2 with IAM role-based access and security group firewall rules
PIPELINE FLOW
Git Push→ Webhook→ Jenkins→ Maven Build→ Nexus→ Tomcat / EC2

// Docker · Jenkins · AWS EC2
DOCKERIZED
DEPLOYMENT PIPELINE

A containerized delivery pipeline that packages the application into Docker images, pushes to DockerHub, and deploys containers on AWS EC2 with full health monitoring.

DockerDockerfileDocker ComposeDockerHubJenkinsAWS EC2
↑ 30% faster environment setup, zero config drift
// What I Built & How It Works
  • Wrote a custom Dockerfile using FROM, COPY, RUN, EXPOSE, and CMD instructions to package the app and all its dependencies into a single portable image
  • Jenkins pipeline automatically builds the Docker image, tags it with version numbers, and pushes it to DockerHub registry on every commit
  • Containers are deployed on AWS EC2 with port mapping and container health checks — if a container crashes, it's detected immediately
  • Used Docker Compose to manage multi-container setup — web app and database running together with isolated networking and service discovery
  • Eliminated "works on my machine" problem — identical container runs on dev, staging, and production environments
CONTAINER FLOW
Code→ Dockerfile→ Docker Image→ DockerHub→ EC2 Container

// Kubernetes · Minikube · AWS EC2
KUBERNETES
ZOMATO WEB APP

Deployed a real-world web application (Zomato clone) on Kubernetes with production-grade features: self-healing pods, zero-downtime rolling updates, and replica management.

KubernetesMinikubekubectlYAML ManifestsDockerAWS EC2NodePort
✓ Zero-downtime deployment with self-healing pods
// What I Built & How It Works
  • Wrote YAML Deployment manifests defining the desired state — Kubernetes constantly ensures the actual state matches it (self-healing)
  • Configured ReplicaSet with 3 replicas — if any pod crashes, Kubernetes automatically brings a new one up within seconds, keeping the app alive
  • Implemented Rolling Update strategy — new version is deployed pod-by-pod, so there's never a moment where the entire app is down during updates
  • Set up NodePort Service for external traffic routing, and debugged kube-proxy, DNS, and inter-pod communication issues during cluster setup
  • Ran Minikube cluster on AWS EC2 with proper security group configuration and monitored pod health using kubectl logs and describe

AWS CLOUD INFRA

TerraformAnsibleAWS EC2S3VPCIAMSubnetsBash Scripts

// Docker Compose · AWS EC2 · GitHub
MULTI-CONTAINER
MICROSERVICES

A production-style multi-tier microservices application with separate containers for web app, database, and reverse proxy — all orchestrated via Docker Compose on AWS.

Docker ComposeDockerAWS EC2GitHubShell ScriptingNetworking
✓ Isolated, independently scalable services
// What I Built & How It Works
  • Designed a docker-compose.yml defining three separate services — web app (port 3000), database (port 5432), and Nginx reverse proxy (port 80) — each in its own container
  • Configured custom Docker networks for inter-container communication — services can talk to each other by name, while being isolated from external access
  • Used Docker volumes to persist database data — even if the database container is restarted, no data is lost
  • Each service can be scaled independently — scale the web app to 5 containers without touching the database, just by changing a number in compose file
  • Deployed the entire stack on AWS EC2 with shell scripts to automate startup, shutdown, and health checking of all services
